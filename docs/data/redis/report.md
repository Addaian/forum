# The rdbLoadObject Question: Ship Cost vs. a 1,500-Line Dispatcher

This audit surfaced a single high-priority decision point, and it lands squarely on the team's two highest-weighted concerns: **velocity** and **simplicity**. The function in question — `rdbLoadObject` in `src/rdb.c`, lines 2406–3894 — carries a measured cyclomatic complexity of 300, twenty times the project threshold of 15. On its face that number reads as classic structural debt. The panel split 6-4 in that direction. But the judge overrode the majority, and the override matters: the verdict frames how much iteration drag this function actually imposes on the team versus how much would be created by attacking it. The headline finding is that this is a parser/dispatcher whose complexity is inherent to the RDB wire format — not runaway accretion. The cost of *living with it* and the cost of *refactoring it* both deserve a clear-eyed look before anyone opens a branch.

## src/rdb.c::rdbLoadObject — the RDB type dispatcher

**Verdict: JUSTIFIED VIOLATION**

The function spans 1,488 lines and dispatches on `rdbtype` across every Redis on-disk type — `RDB_TYPE_STRING`, `RDB_TYPE_LIST`, `RDB_TYPE_SET`, and so on through the full enumeration. The Layer 1 metrics are unambiguous: `complexity=300`, `blast_radius=1.0`, `principle_severity=1.0`, `pattern_violation=1.0`. Lizard's measurement is correct. What the metric does not capture is the *shape* of that complexity. Looking at the snippet, the branching is not nested decision logic that compounds — it is a flat sequence of `if (rdbtype == X) … else if (rdbtype == Y) …` arms, each one self-contained, each one corresponding to exactly one serialization format the system must support to remain wire-compatible with existing RDB files. Every supported Redis type contributes exactly one branch. You cannot reduce the branch count without dropping a type, and dropping a type breaks persistence compatibility with every existing deployment.

This is the case the principle's own documentation carves out: a parser or dispatcher where the complexity is the direct mapping of an external format onto code. The panel's empirical_skeptic cell made the point cleanly — the 300 isn't 300 *interacting* decisions, it's 300 decisions arranged as a switch. The ergonomics_advocate cell carried the simplicity argument that matters to this team specifically: extracting per-type bodies into helper functions does not reduce cognitive load, it merely scatters it. A reader trying to understand "how does Redis load an object" now has to chase pointers across a file or across files to reassemble the dispatch in their head. The indirection is real; the simplification is not.

The `recency` signal is `0.0`. This function is not drifting. It is not accreting new branches on a hot iteration cycle. It is stable, load-bearing, well-understood scaffolding around a stable on-disk format. For a team that weights **velocity** at 1.50 and **simplicity** at 1.20, that combination — stable, inherent, flat dispatch — is exactly the profile of code you leave alone. The PR cycle cost of touching it is high (any change to a 1,488-line function in the persistence path demands serious review), and the iteration drag it currently imposes is low (nobody is editing it weekly; it changes when a new type is added, which is rare).

**The dissent deserves a clear seat at the table.** The strongest counter-argument from the debt-voting cells was not procedural — it was about correctness. A function with cyclomatic complexity 300 has, by definition, on the order of 300 independent execution paths. Exhaustively testing the error paths and edge cases of those branches is practically impossible, and this is the core persistence loader: a latent bug in one of those branches is an incident vector that corrupts data on load or silently mis-deserializes. The "inherent complexity" defense explains *why* the branches exist; it does not explain *how* they are validated. For a team that also weights correctness at 1.00, this is not a footnote. It is the reason the panel split 6-4 rather than landing cleanly on justification.

The judge's response to that concern is structural: the same untested-branch risk applies to a refactored dispatch table just as much as to the current monolith. Splitting 300 branches across 20 helper functions does not, on its own, make those branches more tested. Test coverage of error paths is a separate workstream from extraction, and conflating the two is how teams spend a quarter of their shipping velocity on a refactor that doesn't actually close the risk it was sold against.

**The judge's recommended action**, should the team choose to act despite the JUSTIFIED VIOLATION verdict, is specific: extract the per-type load logic out of `rdbLoadObject` in `src/rdb.c` into a static dispatch table of function pointers — one handler per `RDB_TYPE_*` constant — and reduce the monolithic if-else chain to a single indexed dispatch call. The size estimate is roughly 200–300 lines of new handler functions plus a ~20-line dispatch table. Behavior preservation is mandatory; the win is that each type's loader becomes independently testable.

For this team's value vector, the honest read of that recommendation is that it is a **medium-sized refactor with real ship cost and a marginal simplicity dividend**. The dispatch table is genuinely cleaner than the if-else chain — a 20-line table of `{RDB_TYPE_STRING, loadString}` pairs is the kind of structure where a reader can see the whole shape of the system on one screen. That is a simplicity win. But each of the 20-odd extracted handlers will still carry its own internal branching (length checks, encoding variants, sanitization paths), so the per-handler cognitive load is not trivially small either. The extraction makes the *outer* structure simpler at the cost of fragmenting the *inner* logic across the file.

Given the JUSTIFIED VIOLATION verdict and the team's velocity-first posture, the defensible position is: **do not open a branch for this refactor right now**. The function is stable (`recency=0.0`), the complexity is inherent to the format, and the cells that argued most clearly in the team's vocabulary — ergonomics_advocate on simplicity, empirical_skeptic on the inherence of the dispatch — both landed on the justified side. The override is doing real work here; it is telling you that the lizard score is measuring the wrong thing for this specific function.

What the dissent's correctness concern *does* justify, separately and at lower cost, is a targeted **test-coverage audit of the error paths within `rdbLoadObject`** — specifically the per-type sanitization and length-validation branches that fire when `deep_integrity_validation` is true. That work can be done without touching the function's structure, ships incrementally, and addresses the actual risk surface the debt-voting cells identified. It costs a fraction of the extraction refactor and closes the real failure mode rather than reshuffling it.

If, at some future point, a new persistence type needs to be added and the team finds themselves resisting the addition because the function is becoming hard to extend safely, *that* is the trigger to revisit. The `recency` signal will have moved off zero, the inherent-complexity argument will have weakened (because new types mean the format is no longer stable), and the dispatch-table refactor will pay for itself by making the next addition routine. Until then, the JUSTIFIED VIOLATION verdict means: noted, accepted, not actionable as structural work.

## What to do this quarter

Two things, in order. First, **leave `rdbLoadObject` structurally alone**. The verdict is JUSTIFIED VIOLATION and the override is well-reasoned for a team that weights velocity and simplicity highest; the refactor would consume meaningful PR cycle time and deliver a simplicity dividend that is real but modest, while not addressing the underlying correctness concern the dissent raised. Second, **fund a focused test-pass over the error and sanitization branches** inside `rdbLoadObject` (particularly the `deep_integrity_validation` paths and the per-type length checks). That work answers the dissent's strongest objection directly, ships in small increments, and does not require touching the dispatch structure. Revisit the extraction question only if a new RDB type addition forces the issue.
